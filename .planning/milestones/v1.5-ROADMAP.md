## Phase Details

### Phase 16: 全局元数据体系

**Goal:** 所有采集事件强制携带标准化元数据，支持 OCSF 标准映射

**Depends on:** None (first phase)

**Requirements:** GM-01, GM-02

**Success Criteria** (what must be TRUE):
1. 用户配置数据源时必须提供 vendor_name / product_name / device_type，否则无法保存
2. 用户配置数据源时必须提供 tenant_id 和 environment（prod/dev/test），否则无法保存
3. 系统自动将设备类型映射到 OCSF target_category_uid / target_class_uid
4. 系统自动校验 OCSF 格式合规性，不合规事件拒绝接入并返回错误原因
5. 模板级别的元数据可以覆盖全局默认值

**Plans:** 3 plans in 2 waves

Plans:

- [ ] 16-02-PLAN.md — Wave 1: 创建 src/collection/metadata.py (MetadataEnricher + OCSFMapper)
- [ ] 16-03-PLAN.md — Wave 2: 扩展 ingestion_endpoints.py (metadata 验证 + OCSF 重算 + v1.0 迁移)

### Phase 17: 多渠道采集后端

**Goal:** 支持 Kafka Topic 订阅、Webhook 接收、REST API / 数据库定时轮询三种采集渠道

**Depends on:** Phase 16

**Requirements:** MC-01, MC-02, MC-03

**Success Criteria** (what must be TRUE):
1. 用户可以配置外部 Kafka Topic 订阅，指定 Consumer Group、认证方式（PLAINTEXT/SSL/SASL）、Offset 策略（earliest/latest）
2. 用户可以配置 Webhook 接收端点，系统返回可配置的 HTTP POST URL，支持 Header Secret 校验和 IP 白名单
3. 用户可以配置 REST API 轮询任务，指定轮询间隔、认证方式（OAuth/Token/Basic）、增量游标字段
4. 用户可以配置数据库 JDBC 轮询任务，支持 MySQL/PostgreSQL/达梦/openGauss/TiDB/Kingbase，游标持久化到 Redis
5. 所有渠道事件写入 Kafka raw-events Topic 前自动注入 Phase 16 配置的元数据

**Plans:** 3 plans in 2 waves

Plans:

- [ ] 16-02-PLAN.md — Wave 1: 创建 src/collection/metadata.py (MetadataEnricher + OCSFMapper)
- [ ] 16-03-PLAN.md — Wave 2: 扩展 ingestion_endpoints.py (metadata 验证 + OCSF 重算 + v1.0 迁移)

### Phase 18: 死信队列机制

**Goal:** 解析失败日志自动路由到 DLQ，支持重试和 MinIO 归档，不丢失任何事件

**Depends on:** Phase 17

**Requirements:** SM-01

**Success Criteria** (what must be TRUE):
1. 解析失败的事件自动路由到 dlq-events Topic，不阻塞正常事件处理
2. DLQ 事件自动重试 3 次，间隔 1min / 5min / 30min
3. 超过重试上限的 DLQ 事件自动归档到 MinIO，保留原始日志和时间戳
4. DLQ 大小超过阈值时自动发送管理员告警
5. 管理员可以查看 DLQ 列表，选择事件重新处理或标记为已解决

**Plans:** 3 plans in 2 waves

- [ ] 16-03-PLAN.md — Wave 2: 扩展 ingestion_endpoints.py (metadata 验证 + OCSF 重算 + v1.0 迁移)

### Phase 19: 采集可观测性监控

**Goal:** 暴露采集管道健康指标，支持 Prometheus 抓取

**Depends on:** Phase 17

**Requirements:** SM-02

**Success Criteria** (what must be TRUE):
1. 系统暴露 /metrics/collection Prometheus 端点
2. EPS（每秒事件数）指标准确反映 in/out 双向计数
3. 采集延迟指标（collection_lag_ms）反映事件产生到入 Kafka 的时间差
4. 解析成功率指标（parse_success_rate）反映解析成功比例
5. DLQ 大小指标（dlq_size）和数据源健康状态（datasource_health）可供监控告警
