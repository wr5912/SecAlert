# SecAlert 项目调研材料

## 一、痛点与难点

### 数据层面
- **异构性**：各种品牌/型号的安全设备，日志格式各不相同
- **数据孤岛**：设备间缺乏关联，形成信息孤岛
- **海量数据**：日志量巨大，难以人工分析
- **时效性差**：从告警发生到分析人员看到有延迟，攻击者可能已完成横向移动
- **误报率高**：大量误报导致告警疲劳
- **上下文缺失**：单条告警信息孤立，难以判断真实威胁全貌

### 分析层面
- **缺乏威胁情报**：没有外部情报输入，无法判断告警是否与已知威胁相关
- **攻击链建模难**：将零散的告警还原成完整的攻击链路需要专业知识
- **溯源困难**：日志存储不完整或格式不统一，溯源时证据链断裂

### 运营层面
- **人才稀缺**：专业安全分析师培养周期长，流失率高
- **经验难以传承**：依赖个人经验，人员变动后能力断层
- **合规压力**：等保/GDPR等合规要求与实际安全运营存在落差

### 技术层面
- **加密流量检测**：TLS/SSH等加密通道让传统明文检测失效
- **内网横向检测**：边界防护难以发现已经进入内网的攻击者
- **零日攻击**：基于签名的检测对未知威胁基本无效

## 二、技术解决方案

| 痛点/难点 | 关键技术 |
|-----------|---------|
| 数据异构性 | 日志归一化（ETL/数据管道，转换为 OCSF 统一格式） |
| 数据孤岛 | 实体对齐（通过 IP、用户名、主机名等维度关联不同设备告警）；时序分析（基于时间窗口还原攻击链） |
| 海量数据 | 告警收敛（同源事件聚合，减少重复告警）；流式处理（Kafka+Flink 等实时处理架构） |
| 时效性差 | 实时流式处理架构；自动化告警管道 |
| 误报率高 | 基于统计的异常检测（PCA、信息熵）；告警优先级评分（Risk Scoring） |
| 上下文缺失 | 上下文丰富（WHOIS、漏洞库、资产信息补充告警） |
| 缺乏威胁情报 | 威胁情报匹配（与已知恶意 IP/域名/Hash 进行关联）；ATT&CK 框架映射 |
| 攻击链建模难 | 图关联分析（将告警映射到知识图谱）；攻击链可视化还原 |
| 溯源困难 | 日志完整性保障；溯源链自动构建 |
| 人才稀缺 | SOAR 自动化剧本；人机协作模式（机器处理简单事件，复杂事件交给分析师） |
| 经验难以传承 | 专家规则库化；案例库与知识库建设 |
| 加密流量检测 | TLS 指纹分析；JA3/JA4 指纹匹配；机器学习流量分类 |
| 内网横向检测 | UEBA（用户实体行为分析）；基线偏离检测；微隔离策略 |
| 零日攻击 | 异常行为检测；沙箱分析；威胁狩猎（Threat Hunting） |

## 三、日志格式自动识别技术

### 3.1 问题背景

在完全未知异构多源日志的情况下，如何将告警分析系统接入并进行解析，是首要技术挑战。核心思路是**解耦采集与分析层**，让系统具备"未知日志适配能力"。

### 3.2 技术方案

#### 分层架构

```
[异构数据源] → [采集/适配层] → [标准化格式] → [分析层]
```

关键在于引入**标准化中间格式**（如 OCSF），分析层只依赖统一格式，与具体设备解耦。

#### 适配层核心能力

| 能力 | 说明                                   |
|------|--------------------------------------|
| **格式自动识别** | 优先解析 JSON/OCSF/XML 等自描述格式；无结构日志用正则推断 |
| **字段映射配置** | 提供配置界面，让运维人员手动映射源字段 → 标准字段           |
| **插件化解析器** | 每个设备/厂商对应一个解析插件，按需加载                 |
| **Schema 自动推断** | 对于未知格式，用统计学习推断字段类型和含义                |

### 3.3 核心技术路线

#### 1. 基于规则的方法

| 方法 | 原理 | 优点 | 缺点 |
|------|------|------|------|
| 正则表达式 | 手工编写解析规则 | 速度快、确定性高 | 设备变更时需手动更新 |
| LCS (最长公共子序列) | 通过词频统计和编辑距离聚类 | 无需先验知识 | 多行日志支持差 |

**代表工具**: Spell、MoLFI

#### 2. 基于聚类的方法

| 方法 | 原理 | 优点 | 缺点 |
|------|------|------|------|
| 前缀树聚类 (Drain) | 固定深度前缀树分组相似日志 | 速度快（100万行/秒） | 对高频变量字段处理弱 |
| 层级聚类 (LKE/LogMine) | 层次聚类生成模板 | 准确度较高 | 计算开销大 |

**代表工具**: Drain、LogSig、LogMine

#### 3. 基于深度学习的方法

| 方法 | 原理 | 优点 | 缺点 |
|------|------|------|------|
| Seq2Seq 模型 | 将日志行映射为正则表达式模板 | 端到端学习 | 需要大量标注数据 |
| Transformer | 注意力机制捕获长距离依赖 | 准确度高 | 资源消耗大 |

**代表工具**: DeepLog、LogPPT、LILAC

#### 4. 基于大语言模型 (LLM) 的方法

| 方法 | 原理 | 优点 | 缺点 |
|------|------|------|------|
| Few-shot Prompting | 输入示例日志，LLM 生成解析规则 | 接近人类准确率（F1≈0.99） | 推理成本高、偶发幻觉 |
| RAG (检索增强) | 提供相似历史模板作为上下文 | 减少幻觉、提高准确性 | 需要维护模板库 |
| **DSPy (签名-编译-优化)** | 签名驱动任务定义，MIPRO编译优化，反馈自迭代 | 更可控、可优化、可自进化 | 需要训练数据 |

**代表工具**: Matryoshka（LLM生成正则）、LogSentinelAI、LILAC

**DSPy 方案核心设计**：

```python
# 签名定义：输入原始日志，输出解析配置
class LogParserSignature(dspy.Signature):
    raw_logs = dspy.InputField(desc="3-5条原始日志示例")
    source_type = dspy.InputField(desc="数据源类型描述")

    regex_pattern = dspy.OutputField(desc="Python正则，包含命名捕获组")
    field_mappings = dspy.OutputField(desc="字段映射字典")
    confidence = dspy.OutputField(desc="置信度 0-1")

# 自迭代优化流程
反馈收集 → 训练数据 → MIPRO编译 → 优化后的模型
```

### 3.4 方案选型建议

| 场景 | 推荐方案 |
|------|----------|
| 实时流处理（低延迟） | Drain（规则聚类）+ 正则执行 |
| 离线大规模解析 | LogMine / LKE（层级聚类） |
| 未知格式适配（零样本） | **DSPy + Qwen3-32B（已离线部署）** |
| 生产环境（成本敏感） | 混合方案：规则预过滤 → DSPy 精细化 → 解析器缓存 |

### 3.5 Schema Drift 管理

日志格式会随设备升级而变化，需要持续监控：

| 挑战 | 检测/缓解策略 |
|------|--------------|
| Schema 漂移 | 与基线持续对比，自动识别新增/变更字段 |
| 静默丢失 | 监控 ingest-to-export 计数器、队列深度 |
| 多行/混合日志 | 增量聚类检测概念漂移 |

### 3.6 开源/商业方案对比

| 方案 | 类型 | 核心技术 | SIEM 集成方式 |
|------|------|----------|---------------|
| **Matryoshka** | 开源 | LLM生成正则 | JSON输出，可接入任意SIEM |
| **LogSentinelAI** | 开源 | LLM+Pydantic声明式提取 | 直推Elasticsearch |
| **DSPy** | 开源 | 签名-编译-优化，自迭代 | 自研集成 |
| **Elastic Automatic Import** | 商业 | LLM生成Ingest Pipeline | 原生集成Elastic |
| **IBM QRadar DSM** | 商业 | 450+预置解析模块 | Syslog/API直采 |
| **FortiAnalyzer** | 商业 | 内置+插件市场 | Syslog/HTTPS转发 |

## 四、数据存储与组织架构

### 4.1 数据分层模型

为支撑日志告警分析与攻击链还原，采用四层数据架构：

```
┌─────────────────────────────────────────────────────────┐
│  原始层 (Raw Layer)                                      │
│  - 原始日志全文保留                                       │
│  - 用途：合规审计、规则回溯、重新解析                      │
└─────────────────────────────────────────────────────────┘
                          ↓ 解析
┌─────────────────────────────────────────────────────────┐
│  标准化层 (Normalized Layer)                              │
│  - 统一格式存储（OCSF/ECS 兼容）                          │
│  - 提取通用字段：时间戳、来源IP、目标IP、用户、事件类型等   │
└─────────────────────────────────────────────────────────┘
                          ↓ 丰富
┌─────────────────────────────────────────────────────────┐
│  丰富层 (Enriched Layer)                                  │
│  - 接入威胁情报（IP/域名信誉）                            │
│  - 接入资产信息（主机名、责任人、业务系统）                │
│  - 接入漏洞数据（CVE、受影响资产）                        │
│  - ATT&CK 战术/技术标记                                   │
└─────────────────────────────────────────────────────────┘
                          ↓ 构建
┌─────────────────────────────────────────────────────────┐
│  分析层 (Graph Layer)                                     │
│  - 实体关系图（IP ↔ 用户 ↔ 主机 ↔ 进程）                │
│  - 时序链（同一会话的事件序列）                           │
│  - 攻击链（因果关联的事件链）                             │
└─────────────────────────────────────────────────────────┘
```

### 4.3 实体关系图 (Graph Model)

用于表达复杂关联关系，支撑攻击路径追溯：

**节点类型**
| 节点 | 说明 |
|------|------|
| IP | IP 地址 |
| User | 用户账户 |
| Host | 主机/服务器 |
| Process | 进程 |
| Domain | 域名 |
| File | 文件 |

**边关系（典型攻击场景）**
| 源节点 | 关系 | 目标节点 |
|--------|------|----------|
| 外部IP | 扫描 → | 多个内网IP |
| IP | 暴力破解 → | 认证服务 |
| User | 登录 → | Host |
| Host | 运行 → | Process |
| Process | 连接 → | C2域名 |
| User | 下载 → | 恶意文件 |

## 五、攻击链还原技术

### 5.1 核心关联技术

| 技术 | 作用 |
|------|------|
| **时序窗口关联** | 同一实体（IP/用户）在短时间内的连续异常行为 |
| **因果关系推理** | 基于规则推断事件间的因果（如：认证失败 → 认证成功 → 提权） |
| **图遍历** | 从初始入口点沿关系边追溯攻击路径 |
| **模板匹配** | 匹配已知攻击链模式（如：勒索软件典型行为序列） |

### 5.2 ATT&CK 驱动分析

```
检测 → 映射到 ATT&CK 战术/技术
         ↓
    战术序列还原
    [初始访问] → [执行] → [持久化] → [横向移动] → [数据窃取]
         ↓
    时间线可视化展示
         ↓
    攻击阶段评分（每阶段的风险累积）
```

### 5.3 攻击链序列规则示例

```yaml
# 暴力破解 → 成功登录 → 提权扫描
sequence_rules:
  - name: "暴力破解后成功登录"
    steps:
      - type: auth_failure
        count: ">5 in 5min"
      - type: auth_success
        condition: "same user/IP after step 1"
      - type: internal_scan or privilege_escalation
        condition: "within 30min of step 2"
    result:
      attack_chain: "[初始访问] → [持久化] → [横向移动]"
```

### 5.4 攻击链可视化方案

#### 攻击路径图

```
[外部IP] ──SSH──→ [DMZ服务器] ──RDP──→ [域控制器]
   │              │                    │
   └── 暴力破解 ────┘                    │
                                         │
                            ┌───────────┴───────────┐
                            │   数据窃取 (C2外联)   │
                            └───────────────────────┘
```

#### 交互式调查界面

| 区域 | 内容 |
|------|------|
| 左侧 | 事件列表（按时间排序） |
| 中间 | 事件详情 + 字段解析 |
| 右侧 | 关联实体（同一IP/用户的所有事件） |
| 底部 | 攻击链阶段指示器 |

### 5.5 存储技术选型

| 需求 | 推荐存储 |
|------|----------|
| 原始日志存储 | 对象存储 (S3/MinIO) + 压缩 |
| 标准事件存储 | Elasticsearch / ClickHouse |
| 实体关系存储 | Neo4j / NebulaGraph（图数据库） |
| 快速检索 | Elasticsearch（全文索引） |
| 聚合分析 | ClickHouse / Elasticsearch Rollup |

## 六、产品交互设计

### 6.1 数据接入用户体验

#### 配置方式

| 方式 | 适用场景 | 用户操作 |
|------|----------|----------|
| **零配置（AI 自动识别）** | 完全未知的日志格式 | 提供 3-5 条示例日志，系统自动识别并推荐字段映射 |
| **设备模板库** | 已知设备（主流品牌） | 选择设备型号，一键应用模板 |
| **可视化字段映射** | 自动识别不准确时 | 点选/拖拽式映射，实时预览效果 |
| **批量接入** | 多台同类型设备 | 批量导入设备列表，统一应用模板 |

#### 接入流程

```
Step 1: 选择接入方式
  ○ 我知道设备型号（选模板）
  ○ 不知道，我提供示例日志（AI 自动识别）
  ○ 我想手动配置（可视化映射）

Step 2: 配置 / 确认
  系统展示：识别结果 + 字段映射
  用户：确认 或 调整

Step 3: 测试 & 生效
  用历史日志测试解析准确率
  达标后开启实时接入
```

### 6.2 设计原则

| 原则 | 说明 |
|------|------|
| **保守默认** | 自动识别结果默认不启用，需人工确认 |
| **透明可解释** | 用户能看懂系统为什么这么识别 |
| **容错友好** | 识别错了容易回退、容易改 |
| **渐进增强** | 第一次用简单模式，高级用户在高级模式挖掘 |

## 七、告警分析技术

### 7.1 分析层次

基于已组织好的数据，告警分析分为三个层次：

```
┌─────────────────────────────────────────────────────────────┐
│  Level 1: 单点检测                                           │
│  规则匹配、签名检测、阈值告警                                 │
└─────────────────────────────────────────────────────────────┘
                          ↓
┌─────────────────────────────────────────────────────────────┐
│  Level 2: 关联分析                                           │
│  时序关联、跨源关联、统计异常                                  │
└─────────────────────────────────────────────────────────────┘
                          ↓
┌─────────────────────────────────────────────────────────────┐
│  Level 3: 高级分析                                           │
│  图分析、AI模型、威胁狩猎                                      │
└─────────────────────────────────────────────────────────────┘
```

### 7.2 Level 1：单点检测（规则匹配）

#### 1.1 签名规则检测

```yaml
# 基于已知攻击模式的规则
rules:
  - name: "暴力破解检测"
    condition: |
      event.type == "auth_failure"
      AND count > 5 WITHIN 5min BY src_ip
    severity: high
    mitre_tactics: [TA0001]

  - name: "C2通信检测"
    condition: |
      event.domain IN威胁情报库.malicious_domains
      OR event.url IN威胁情报库.malicious_urls
    severity: critical
    mitre_techniques: [T1071, T1105]
```

#### 1.2 阈值告警

| 类型 | 规则示例 |
|------|----------|
| 流量阈值 | 同一IP在5分钟内对外发起 >100次连接 |
| 失败次数 | 同一用户登录失败 >3次 |
| 敏感时段 | 非工作时段有管理员操作 |

#### 1.3 IOC 匹配

```
告警事件 → 提取IOC（IP/域名/Hash/URL）
              ↓
         威胁情报库匹配
              ↓
    高置信度恶意 → 直接告警/自动封禁
```

### 7.3 Level 2：关联分析

#### 2.1 时序链关联

同一实体（IP/用户/主机）在时间线上的连续异常行为：

```
时间线分析：
┌─────────────────────────────────────────────────────────────┐
│  10:00  - 外部IP尝试SSH (防火墙允许)                         │
│  10:01  - SSH认证失败 (root, 密码错误) ×5                   │
│  10:02  - SSH认证成功 (root)                                │
│  10:03  - 同一IP扫描内网段                                  │
│  10:10  - 尝试RDP到域控制器                                  │
│  10:15  - 大流量外传                                         │
│                                                             │
│  结论：这是一次典型的 [外部渗透 → 暴力破解 → 横向移动]        │
└─────────────────────────────────────────────────────────────┘
```

#### 2.2 跨设备关联

同一攻击链中不同安全设备告警的关联：

```
┌──────────────┐    ┌──────────────┐    ┌──────────────┐
│  防火墙       │    │  IDS/IPS     │    │  终端EDR     │
│  (边界)       │    │  (流量检测)   │    │  (主机监控)   │
└──────┬───────┘    └──────┬───────┘    └──────┬───────┘
       │                   │                   │
       └───────────────────┼───────────────────┘
                           ↓
                   实体对齐关联
                   (src_ip + timestamp + user)
                           ↓
                   还原完整攻击链
```

#### 2.3 统计异常检测

| 方法 | 说明 | 应用场景 |
|------|------|----------|
| 基线偏离 | 建立正常行为基线，检测偏离 | 异常登录时间、异常流量大小 |
| 信息熵 | 计算字段值的信息熵 | 随机生成的C2域名、随机密码 |
| PCA | 主成分分析降维检测异常 | 多维特征异常（如网络连接模式） |

#### 2.4 告警收敛

同源事件的聚合，减少告警疲劳：

```
触发点                收敛规则              输出
─────────────────────────────────────────────────────
同一IP 100次扫描  →  聚合为"端口扫描事件"  →  1条告警
同一用户10次失败  →  收敛为"暴力破解尝试"  →  1条告警
同一进程500次行为 →  聚类为"可疑进程活动" →  1条告警
```

### 7.4 Level 3：高级分析

#### 3.1 图关联分析

基于实体关系图进行深度分析：

```
分析方法：
- 社区发现：识别异常聚集的实体群
- 最短路径：查找两个异常实体间的关联路径
- 中心性分析：识别高风险"桥梁"节点
- 异常子图检测：发现与正常模式显著不同的子图

示例：
  发现某主机同时连接了：
  - 已知恶意IP
  - 内部敏感资产
  - C2域名
  → 该主机被攻陷的可能性极高
```

#### 3.2 AI/ML 模型

| 模型类型 | 应用场景 | 输入特征 |
|----------|----------|----------|
| 分类模型 | 告警分类（True Positive / False Positive） | 告警特征、上下文、历史判定 |
| 聚类模型 | 异常告警分组 | 事件时序、类型、关联实体 |
| 异常检测 | 未知攻击发现 | 多维特征基线偏离 |
| NLP模型 | 日志摘要、攻击叙事生成 | 原始日志、告警文本 |

#### 3.3 攻击链模板匹配

预定义攻击链模板，匹配还原的攻击路径：

```
已知攻击模板库：

模板1: 勒索软件典型路径
  [钓鱼邮件] → [宏执行] → [横向移动] → [数据加密] → [勒索通信]

模板2: APT持久化路径
  [初始访问] → [提权] → [持久化] → [C2通信] → [凭据窃取]

模板3: 内部威胁路径
  [登录] → [敏感数据访问] → [批量下载] → [外发]

匹配：当前攻击链 → 与模板匹配度评分 → 输出攻击类型推断
```

#### 3.4 威胁狩猎 (Threat Hunting)

主动搜寻，而非被动等待告警：

```
狩猎流程：
1. 假设生成 (HI：Howells)
   - "如果攻击者已在内网，会留下什么痕迹？"

2. 主动搜索
   - 搜索异常子网段的通信
   - 检查异常时间段的登录行为
   - 寻找稀有进程或计划任务

3. 发现 → 验证 → 响应
```
