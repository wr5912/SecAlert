## 1. 项目概述

### 1.1 项目背景

**SecAlert** 是一款面向企业普通 IT 运维人员的智能网络安全告警分析系统。系统核心价值在于帮助非安全专业背景的运维人员自动过滤海量告警，只呈现真正需要关注的安全威胁。

### 1.2 核心痛点

| 痛点 | 解决方案 |
|------|----------|
| 每天数万条异构安全设备告警 | 三层解析管道（模板→Drain→LLM）自动识别 |
| 误报率高（通常 70%+） | DSPy AI 引擎自动过滤误报，目标 <30% |
| 告警孤岛，无关联分析 | 基于滑动窗口的跨设备攻击链关联 |
| 非专业运维人员难以处置 | 简洁 UI + 明确处置建议 |

### 1.3 核心约束

| 约束 | 要求 |
|------|------|
| **部署模式** | 私有化离线部署，无外部云依赖 |
| **处理能力** | 每天 3 万+ 条告警，延迟可接受 |
| **用户画像** | 非专业运维人员，界面必须极度简单 |
| **AI 推理** | 所有 AI 推理基于私有化 Qwen3-32B |
| **国产化** | 支持达梦(DM)、TiDB、openGauss、Kingbase |

### 1.4 功能范围

### 2.2 分层职责

| 层级 | 组件 | 职责 |
|------|------|------|
| **采集层** | Vector + Logstash | 统一数据采集，协议转换，多渠道采集 |
| **消息层** | Kafka | 事件流缓冲，解耦 |
| **解析层** | Parser Pipeline | 三层解析，格式标准化 |
| **存储层** | ES/Neo4j/ClickHouse/MinIO | 多模态存储 |
| **分析层** | Rule Engine / Graph / AI | 关联分析，AI 分类 |
| **API 层** | FastAPI | REST 接口 + WebSocket + Prometheus Metrics |
| **前端层** | React | 用户界面 |
| **AI 助手层** | AIPanel / AgentClient / MCP | 上下文感知智能辅助，跨层协作 |
| **可观测性层** | Prometheus / DLQ | 指标监控，死信队列处理 |

### 2.3 架构模式

采用 **分层流处理架构** (Layered Stream Processing Architecture):

1. **采集层** 负责从各类数据源获取原始数据
2. **消息层** 提供缓冲和削峰能力
3. **解析层** 实现格式标准化和归一化
4. **存储层** 支持多种存储引擎的读写
5. **分析层** 进行规则检测、图关联、AI 推理
6. **API 层** 提供统一接口
7. **可观测性层** 提供指标监控和死信处理
7. **前端层** 呈现用户交互界面

class ParserRegistry:
    """解析器注册表，支持版本管理"""

    def __init__(self):
        self._parsers: Dict[str, Parser] = {}
        self._templates: Dict[str, Template] = {}

    def register_template(self, name: str, template: Template):
        """注册解析模板"""
        self._templates[name] = template

    def get_parser(self, device_type: str) -> Parser:
        """获取解析器"""
        return self._parsers.get(device_type)

    def match_template(self, raw_log: str) -> Optional[Template]:
        """模板匹配"""
        for template in self._templates.values():
            if template.matches(raw_log):
                return template
        return None

# src/analysis/service.py
class AnalysisService:
    """AI 分析服务"""

    def __init__(self, llm_client):
        self.classifier = DSPyClassifier()
        self.metrics = MetricsCalculator()

    async def analyze_alert(self, alert: Alert) -> AnalysisResult:
        """分析单条告警"""
        # 1. DSPy 分类
        classification = await self.classifier.classify(alert)

        # 2. 计算指标
        metrics = self.metrics.calculate(alert, classification)

        # 3. 生成处置建议
        remediation = await self._generate_remediation(alert, classification)

        return AnalysisResult(
            is_fp=classification.is_false_positive,
            severity=classification.severity,
            confidence=classification.confidence,
            remediation=remediation,
            metrics=metrics
        )

    async def analyze_batch(self, alerts: List[Alert]) -> BatchResult:
        """批量分析"""
        results = await asyncio.gather(*[self.analyze_alert(a) for a in alerts])
        return BatchResult(results=results)

```python
class AlertQueryBuilder:
    """构建告警过滤查询条件 (ES-03)"""

    DEFAULT_FILTERS = {
        "time_range": "now-1h~now",
        "severity": ["high", "critical"],
        "risk_score": {"gte": 70}
    }

    def build_es_query(self, filters: dict = None) -> dict:
        """构建 Elasticsearch 查询"""
        f = {**self.DEFAULT_FILTERS, **(filters or {})}
        return {
            "query": {
                "bool": {
                    "must": [
                        {"range": {"@timestamp": {"gte": "now-1h", "lte": "now"}}},
                        {"terms": {"severity": ["high", "critical"]}},
                        {"range": {"risk_score": {"gte": 70}}}
                    ]
                }
            }
        }
```

**告警过滤查询条件 (ES-03):**
- 时间范围: `now-1h` ~ `now`
- 严重度: `high` 或 `critical`
- 风险评分: `risk_score >= 70`

  step: 1,
  deviceType: null,
  connection: null,
  logFormat: null,
  customRegex: null,

};

export const useIngestionStore = create<IngestionState>((set) => ({

## 14. 团队协作策划

### 14.1 核心理念转变

| 传统模式 | Claude Code 模式 |
|----------|-----------------|
| 3 人串行排队开发 | 3 个 Worktree **并行**工作 |
| 把 AI 当"高级自动补全" | 把 AI 当"初级并行开发团队" |
| 单一代码库来回切换 | 每个任务独立分支 + 隔离环境 |
| 人类发现问题后直接手动修复 | 人类发现问题，AI 分析并修复 |

**核心原则：不要让 Claude 串行排队干活，而是使用 Git Worktrees 并行处理。**

**人机协作边界：人类负责发现问题，AI 负责分析问题并修复。** 在使用 Claude Code 进行编码过程中，即使你（人类）发现了问题，也不要直接手动修改代码。正确做法是：先去调研修复问题的最佳实践，将问题现象描述清楚，然后要求 Claude Code 按照最佳实践自行修复。手动修改容易忽略关联影响，而 AI 可以基于完整上下文分析问题的根因和连锁反应，给出更安全的修复方案。

### 14.5 三大避坑指南

#### 坑1: 上下文爆炸导致"间歇性失忆"

| 症状 | 原因 |
|------|------|
| Claude 开始忘记早期指令 | 同一 Session debug 太长时间 |
| 代码越改越乱 | 上下文窗口耗尽 |

**避坑方案:**
```markdown
1. 高频重启: 把 Claude 当"每天早上都会失忆但极快的员工"
   - 每次完成**闭环**小任务，立刻 exit

2. 隔离调查: 开专门 worktree 只用于让 Claude 读日志、查问题
   - 查明原因后，新开会话执行修复

3. 模型降级:
   - 简单文本/配置修改 → Haiku
   - 日常开发 → Sonnet
   - 大架构梳理 → Opus
```

#### 坑2: AI 盲目自信，大项目中越改越乱

| 症状 | 原因 |
|------|------|
| 修复一个 Bug 引出三个新 Bug | Claude 写出"幻觉代码" |
| 看起来完美但跑不通 | 缺乏验证闭环 |

**避坑方案:**
```markdown
1. 限制 Scope: 明确指出修改范围
   - "参考 src/widgets/ 目录，只修改 foo.py"

2. 提供验证闭环 (最关键): 永远不要让人成为 AI 唯一的反馈环
   - 要求: "改完后运行 `pytest tests/ -v`，确保测试通过后再结束"
```

#### 坑3: 团队成员权限环境不一致

| 症状 | 原因 |
|------|------|
| 有人使用 `--dangerously-skip-permissions` | 权限配置混乱 |
| AI 误删文件 | 安全设置不当 |

**避坑方案:**
```markdown
1. 精细化配置安全白名单
   - 允许自由运行: `npm test`, `git status`
   - 禁止危险操作: `rm -rf`, `git push --force`

2. 统一配置文件
   - 将 `.claude/settings.json` 提交 Git
   - 确保三人环境一致
```
